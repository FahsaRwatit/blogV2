# Harness Engineering：2026 年最重要的新工程学科

> "Agent 不难，难的是 Harness。"
> —— AI 工程领域 2026 年最广为流传的一句话

---

## 一切从一行代码开始

2026 年初，OpenAI Codex 团队宣布了一件让整个工程圈震惊的事：他们构建了一个超过 **100 万行代码**的生产级应用，其中**零行代码由人类编写**。

工程师们没有写代码。他们设计了一套让 AI Agent 能可靠地写代码的**系统**。

这套系统，就是 **Harness**。

而设计和维护这套系统的人，就是 **Harness Engineer**。

---

## 什么是 Harness Engineering？

Harness Engineering 是一门新兴的工程学科，专注于围绕 AI Agent 构建完整的运行环境，包括定义其约束、工具链、反馈循环和生命周期管理系统，确保 Agent 能持续、可靠、安全地执行任务，而不仅仅依赖单次 Prompt 或上下文。

用一个比喻来理解："Harness"这个词来自马具——缰绳、马鞍、马衔，是驾驭一匹强壮但不知方向的马所需的全套装备。AI 模型就像那匹马——强大、快速，但不知道该往哪里跑。Harness 就是把它引向正确方向的那套系统。

---

## AI 工程的三次进化

理解 Harness Engineering，需要先理解它从哪里来：

AI 交互经历了三个明显的阶段：**Prompt Engineering（2022-2024）**、**Context Engineering（2025）**，以及现在的 **Harness Engineering（2026）**。

**第一阶段：Prompt Engineering**
靠一句好的 Prompt 让 AI 输出想要的结果。单次交互，聪明的措辞就够了。

**第二阶段：Context Engineering**
由 Andrej Karpathy 等人推动的认知转变：一个 Prompt 远远不够，需要动态构建上下文窗口，把相关文档、对话历史、工具定义、RAG 结果一起塞进去。

**第三阶段：Harness Engineering**
这是当前的前沿。它包含了前两个阶段，但在更高的抽象层面运作。不再是优化"邮件的措辞"或"邮件的附件"，而是架构整个"办公室"——Agent 的工作流、约束、反馈循环、工具链和生命周期。

---

## 为什么现在？三个驱动力

三个趋势让 Harness Engineering 在 2026 年成为必要：

**模型商品化**：Claude、GPT-4、Gemini 等模型在标准基准上的表现差距越来越小。模型本身不再是竞争优势，模型周围的系统才决定 Agent 能否在生产中可靠运行。

**Agent 从演示走向生产**：2025 年大多数 Agent 还是 Demo 或概念验证，2026 年组织开始部署处理真实客户交互、编写生产代码、管理基础设施、做出财务决策的 Agent。可靠性要求从"演示令人印象深刻"上升到"不能宕机"。

**基准测试失效**：标准基准衡量单轮任务完成率，但生产 Agent 连续运行数小时甚至数天。单轮测试好的模型，未必能扛住多轮长任务的考验。

---

## 一个数字改变了所有人的认知

2026 年初，两篇文章重新定义了业界对 Agent 质量的理解。LangChain 的 Deep Agents 团队发布了一组数据：他们的编程 Agent 在 Terminal-Bench 2.0 排行榜上从前 30 名之外跃升至第 5 名，提升了 13.7 个百分点——**整个过程中 GPT-5.2-Codex 模型完全没有变动，改变的只有系统 Prompt、工具定义和中间件**。

这个数字意味着什么？

**模型是一半，Harness 是另一半。不在模型里的所有东西，都在 Harness 里。**

---

## Harness 到底包含什么？

一个完整的 Harness 通常包含以下核心组件：

- **编排循环**（ReAct 或 Thought-Action-Observation 循环）
- **系统 Prompt 组装**
- **工具定义与工具调用执行**
- **模型和工具调用前后的 Hook 和中间件**
- **子 Agent 委托机制**
- **进程内记忆和状态管理**
- **上下文窗口压缩和卸载**
- **会话生命周期管理**
- **结构化输出解析**
- **自我验证逻辑**

每一个决策都是任务特定或模型特定的。Harness 是领域知识的居所——比如编程 Agent 应该在声明完成前重新阅读自己的 diff，研究 Agent 应该先规划再执行，支持 Agent 在三次工具调用失败后应该升级。

---

## Harness Engineering 的四个核心实践

### 1. 约束设计（Architectural Constraints）

告诉 Agent "遵循我们的编码规范" 和把一个检查规范的 Linter 接入 PR 合并流程，是本质不同的两件事。前者依赖概率性合规，后者强制确定性约束。Harness Engineering 的核心就是把这两者的边界说清楚。

### 2. 反馈循环（Feedback Loops）

Anthropic 的研究揭示了一个关键缺陷：模型无法可靠地评估自己的工作。解决方案是一个受 GAN 启发的系统，分离的 Generator 和 Evaluator Agent——这是健壮 Harness 的核心组件之一。

### 3. 规则文件（Rules Files / AGENTS.md）

规则文件是持久的、仓库范围的指令集，在 Agent 会话开始时自动注入上下文。它与普通 Prompt 的关键区别是：规则文件跨会话持久存在，由运行时自动注入，作用于整个目录树，并可跨父子目录分层组合。

OpenAI 的代码仓库目前使用了 88 个 AGENTS.md 文件，覆盖各个子模块，展示了单仓库规模的约束组合能力。

### 4. 验证与自修复（Verification & Self-Repair）

好的 Harness 不只是防止 Agent 做错事，还让 Agent 知道自己做错了，并能自我纠正。CI 检查、自动测试、Lint 结果都可以成为 Agent 反馈循环的输入，而不是只给人看。

---

## 一个反直觉的结论

METR 2025 年的一项 RCT 研究发现，有经验的开源开发者使用 AI 工具后，平均**慢了 19%**，而他们自认为变快了。Harness 理论恰好解释了这个差距：没有系统包裹 Agent，反复审查、修复、重新 Prompt 的代价会淹没所有收益。

换句话说：**没有好的 Harness，AI 不会让你更快，只会让你更慢。**

---

## 这对软件工程师意味着什么？

Harness Engineering 不是要取代工程师，而是在改变工程师的工作重心：

| 以前 | 现在 |
|---|---|
| 写代码 | 设计让 AI 写好代码的系统 |
| 手动审查每一行 | 设计能自动发现问题的反馈循环 |
| 优化 Prompt 措辞 | 设计跨会话有效的约束架构 |
| 关注单次任务完成 | 关注长时间运行的 Agent 可靠性 |

核心原则始终不变：**问责留在人类这里**。成熟度不是把责任转移给 Agent，而是人类通过把系统建得足够好，来赢得委托更多执行的权利。当 Agent 出了 Bug，依然是人类来承担责任——这个认知应该影响每一个关于 Harness 的架构决策。

## 如何开始？

如果你想开始实践 Harness Engineering，有几个低门槛的切入点：

**第一步：写一个 AGENTS.md**
在你的仓库根目录创建 AGENTS.md，定义 AI Agent 的行为规范——不要做什么、遇到什么情况要停下来问人、代码风格要求等。这是最轻量的 Harness 起点。

**第二步：把 CI 接入反馈循环**
让 Agent 能读到 Lint、测试、构建的结果，而不只是把这些结果给人看。如果 Agent 能看到自己写的代码跑了什么测试、失败在哪里，它才能自我纠正。

**第三步：定义约束边界**
明确 Agent 能做什么、不能做什么，不要靠 Prompt 里的"请注意"，要靠流程上的硬门禁——比如不允许没有测试覆盖的代码合并，无论是谁写的。

**第四步：建立评估机制**
好的 Harness 需要持续度量。不只是"功能上线了吗"，还有"Agent 生成的代码质量、安全性、一致性是否在提升"。

---

## 总结

如果说 2025 年是 AI Agent 证明自己能写代码的一年，2026 年是我们学到"Agent 不难，难的是 Harness"的一年。

Harness Engineering 不是一个工具，不是一个框架，而是一种思维方式的转变：

**从"如何让 AI 做得更好"，到"如何建一个让 AI 持续做好的系统"。**

这是 2026 年最值得每一位软件工程师认真投入的方向。
